

# Fix Cloudflare Pages Build: Bun Lockfile Mismatch

## Root Cause

Cloudflare Pages runs `bun install --frozen-lockfile` with Bun **1.2.15**. Your repo's `bun.lockb` was generated by a newer Bun version, so Cloudflare logs:

```
Outdated lockfile version: failed to parse lockfile: 'bun.lockb'
warn: Ignoring lockfile
error: lockfile had changes, but lockfile is frozen
```

Frozen mode forbids regeneration, so install aborts before the build ever starts.

Additionally, `bunfig.toml` currently has:
```
[install]
saveTextLockfile = false
```
This forces the binary `bun.lockb` format, which is exactly the format Cloudflare's older Bun cannot read.

A `package-lock.json` already exists in the repo (created in a previous step), so npm is a viable, more stable installer for Cloudflare.

## Fix: Switch Cloudflare to Use npm

npm is bundled at a stable version on Cloudflare and reads `package-lock.json` reliably. This avoids fighting Bun version drift.

### Changes

**1. Add `.nvmrc`** (pins Node version Cloudflare uses):
```
22
```

**2. Add `.npmrc`** (avoids peer-dep install failures common with React 19 + TanStack):
```
legacy-peer-deps=true
```

**3. Delete `bunfig.toml`** — removes the `saveTextLockfile = false` setting so any future Bun runs produce the newer text-based lockfile, and signals to Cloudflare's auto-detection to prefer npm.

**4. Delete `bun.lockb`** — eliminates the unreadable binary lockfile. `package-lock.json` becomes the single source of truth.

**5. In the Cloudflare Pages dashboard** (you do this, not a code change):
- **Build command:** `npm run build`
- **Build output directory:** `.output/public` (TanStack Start's Cloudflare target writes here)
- **Environment variables:**
  - `NODE_VERSION` = `22`
  - `NPM_FLAGS` = `--legacy-peer-deps` (belt-and-suspenders if `.npmrc` is missed)

### Why Not "Just Update Bun on Cloudflare"

Cloudflare Pages does not let you pick an arbitrary Bun version per project — it ships whatever Bun is in the build image. Pinning Node + npm is the only durable fix.

### Why Not Cloudflare Workers Instead of Pages

Your `wrangler.jsonc` is set up for Workers (`main: "@tanstack/react-start/server-entry"`), and TanStack Start's Cloudflare target produces a Workers-compatible build. **Workers (via the "Workers Builds" / "Connect to Git" flow) is actually the better fit** for this app because it natively runs the SSR server entry. Pages is primarily static + Functions and will need the same lockfile fix anyway.

If you'd prefer, the alternative path is:
- Cloudflare Dashboard → **Workers & Pages → Create → Workers → Connect to Git**
- Build command: `npm run build`
- Deploy command: `npx wrangler deploy` (auto-detected from `wrangler.jsonc`)
- Same `.nvmrc` / `.npmrc` / lockfile cleanup applies
